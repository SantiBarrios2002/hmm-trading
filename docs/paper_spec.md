# Paper Specification And Ambiguity Table

This document is the review map for the repository's interpretation of
Christensen, Turner, and Godsill, *Hidden Markov Models Applied To Intraday
Momentum Trading With Side Information*. It records where the implementation is
paper-faithful, where it is an engineering approximation, and which paper
components are excluded by the project scope in
[`IMPLEMENTATION_PLAN.md`](../IMPLEMENTATION_PLAN.md).

| Component | Paper says | Repo interpretation | Deviation type | Acceptance risk |
|---|---|---|---|---|
| Observation construction | §7 uses aggregated ES futures returns at 1-minute frequency. | Market data is normalized to `timestamp`, `price`, and optional `volume`, then converted to log returns by [`data.py`](../src/hft_hmm/data.py) and [`preprocessing.py`](../src/hft_hmm/preprocessing.py). | engineering approximation | Data vendor/window mismatch |
| Market-data contract and sampling frequency | §7 reports paper experiments on 1-minute ES futures bars. | The repo supports local Databento ES parquet for paper-replication runs and yfinance/daily data for lightweight development; see Issue 02b, [`data/README.md`](../data/README.md), and [`configs/`](../configs/). | engineering approximation | Fixture scale may not match paper |
| Log-return preprocessing | The model is fitted to return increments `Delta y_t`. | [`compute_log_returns`](../src/hft_hmm/preprocessing.py) computes `ln(p_t / p_{t-1})`; downstream APIs expect cleaned finite returns. | paper-faithful | Low |
| Hidden-state definition and state labels | §3 interprets latent states through their return behavior. | [`state_metadata.py`](../src/hft_hmm/core/state_metadata.py) defines a `StateGrid` ordered by mean return with stable labels for plotting, filtering, and summaries. | evaluation layer | Label names are repo conventions |
| PLR initialization | §3.1 uses piecewise linear regression as an HMM initialization aid. | [`plr_baseline.py`](../src/hft_hmm/models/plr_baseline.py) implements deterministic top-down segmentation and exposes state summaries for optional HMM seeding. | engineering approximation | Exact segmentation differs |
| Emission distribution | §3 models observations with state-dependent Gaussian emissions. | [`GaussianHMMWrapper`](../src/hft_hmm/models/gaussian_hmm.py) fits a one-dimensional diagonal Gaussian HMM and orders states by ascending mean return. | paper-faithful | Backend details differ |
| Baum-Welch EM | §2.3 and §3 fit HMM parameters with Baum-Welch as one estimation route. | The repo delegates EM to `hmmlearn`, with deterministic seeds, stabilized priors, and stored likelihood history in [`gaussian_hmm.py`](../src/hft_hmm/models/gaussian_hmm.py). | engineering approximation | Library implementation differs |
| MCMC parameter estimation | The paper also discusses Metropolis-Hastings estimation. | MCMC fitting is excluded by [`IMPLEMENTATION_PLAN.md §2.5`](../IMPLEMENTATION_PLAN.md#25-scope-exclusions). | excluded by scope | Must be disclosed |
| Variance floor | The paper flags variance collapse on tick-grid data. | The wrapper exposes `min_variance`, defaults it to `1e-8`, and either clamps or raises via `variance_floor_policy`; see Issue 21 and [`gaussian_hmm.py`](../src/hft_hmm/models/gaussian_hmm.py). | engineering approximation | Floor value is defensible, not paper-estimated |
| Baseline transition model | The baseline HMM uses a fixed transition matrix between hidden states. | [`GaussianHMMResult`](../src/hft_hmm/models/gaussian_hmm.py) exposes one row-stochastic transition matrix used by filtering and signals. | paper-faithful | Low |
| Forward filtering | §6 uses filtering probabilities to predict the next return. | [`forward_filter.py`](../src/hft_hmm/inference/forward_filter.py) implements log-space normalized filtering and computes expected next returns from predicted state probabilities. | paper-faithful | Low |
| Trading signal | §8 applies sign-based trading from the expected return signal. | [`strategy/signals.py`](../src/hft_hmm/strategy/signals.py) converts expected returns into long/short/flat signals and aligns them to future returns. | paper-faithful | Alignment risk |
| Signal / realized-return alignment | §8 evaluates signal returns against subsequent realized returns. | [`align_signal_with_future_return`](../src/hft_hmm/strategy/signals.py) applies `signal[t-1] * return[t]` before metrics are computed. | paper-faithful | Off-by-one bugs are high impact |
| Cost model | §8 evaluates trading performance with transaction costs. | [`evaluation/metrics.py`](../src/hft_hmm/evaluation/metrics.py) subtracts linear basis-points-per-turnover costs and reports pre- and post-cost summaries. | evaluation-layer | Cost simplification |
| `K` selection | §4 compares cross-validation, AIC/BIC, and MCMC bridge sampling. | [`selection/model_selection.py`](../src/hft_hmm/selection/model_selection.py) implements AIC/BIC only and documents the parameter count formula. | engineering approximation | Missing CV/bridge sampling |
| MCMC bridge sampling | The paper uses bridge sampling for marginal likelihood comparisons. | Bridge sampling is excluded by [`IMPLEMENTATION_PLAN.md §2.5`](../IMPLEMENTATION_PLAN.md#25-scope-exclusions). | excluded by scope | Must be disclosed |
| Walk-forward scheme | §2.3 estimates parameters using the previous `H` days, then forecasts forward. | [`walk_forward.py`](../src/hft_hmm/experiments/walk_forward.py) defaults to train on 23 days, forecast one day, and assert train/forecast ordering. | paper-faithful | Calendar/session simplification |
| Reproducible YAML configs and run IDs | The paper reports repeatable experiments, but does not define this repo's artifact contract. | [`experiment_config.py`](../src/hft_hmm/config/experiment_config.py), [`runner.py`](../src/hft_hmm/experiments/runner.py), and [`scripts/repro.py`](../scripts/repro.py) serialize configs and use `sha256(resolved_config_yaml)[:12]` run IDs. | evaluation layer | Artifact drift if configs change |
| Volatility-ratio predictor | §4.2 Predictor I is a short/long RiskMetrics EWMA volatility ratio. | [`volatility_ratio.py`](../src/hft_hmm/features/volatility_ratio.py) implements the finite-sum EWMA formula with paper defaults `lambda = 0.79`, `psi_fast = 50`, and `psi_slow = 100`. | paper-faithful | Low |
| Intraday seasonality predictor | §4.2 Predictor II is based on exchange-local time of day. | [`seasonality.py`](../src/hft_hmm/features/seasonality.py) converts UTC timestamps to Chicago time, then emits a fixed-width scalar bucket or normalized coordinate. | engineering approximation | Scalar encoding differs |
| Spline predictor fitting | §4.1 uses spline-estimated conditional means of returns given side information. | [`splines.py`](../src/hft_hmm/features/splines.py) fits deterministic least-squares cubic splines with quantile knots and optional demeaning. | engineering approximation | Exact fitting procedure unspecified |
| Standalone side-information backtests | §4 evaluates predictors before IOHMM comparison. | [`standalone_predictor.py`](../src/hft_hmm/experiments/standalone_predictor.py) runs volatility and seasonality predictors without constructing an HMM state object; see Issue 22. | evaluation-layer | Small samples may show no traction |
| IOHMM transition conditioning | §4 conditions transitions on side information through a side-information model. | [`iohmm_approx.py`](../src/hft_hmm/models/iohmm_approx.py) uses finite side-information buckets and estimates one transition matrix per bucket with smoothing. | engineering approximation | Main model simplification |
| Bucketed transition approximation | Gate H interprets the paper's side-information conditioning through spline-bucketed transition matrices. | [`fit_bucketed_transition_model`](../src/hft_hmm/models/iohmm_approx.py) estimates one row-stochastic `K x K` transition matrix per side-information bucket. | engineering approximation | Bucket boundaries affect results |
| Quantile-boundary follow-up | Issue 42 tracks quantile boundary mode for bucketed IOHMM transitions. | Quantile-derived bucket boundaries are future work only and are not part of the Issue 23 documentation deliverable. | evaluation layer | Follow-up remains open |
| Asynchronous IOHMM | The paper sketches an asynchronous mixed-frequency variant. | Asynchronous IOHMM is excluded by [`IMPLEMENTATION_PLAN.md §2.5`](../IMPLEMENTATION_PLAN.md#25-scope-exclusions). | excluded by scope | Must be disclosed |
| Multi-security evaluation | The paper context includes broader intraday momentum experiments. | The repo is single-security by design; portfolio construction is excluded by [`IMPLEMENTATION_PLAN.md §2.5`](../IMPLEMENTATION_PLAN.md#25-scope-exclusions). | excluded by scope | Scope limitation |
| Production execution | Real trading would require venue, latency, and order-book modeling. | Production execution concerns are excluded; the repo uses simple turnover costs only. | excluded by scope | Scope limitation |
| Figure generation | Gate I requires report-ready figures from scripts, not notebooks. | Existing selection plots live in [`selection/plots.py`](../src/hft_hmm/selection/plots.py); Issue 18 tracks a broader visualization package. | evaluation-layer | Incomplete until Issue 18 |
| Paper comparison results | Gate J requires an honest results-vs-paper table and gap analysis. | Issue 24 will cite reproducible run artifacts and categorize gaps as data, model scope, approximation, or numerical variation. | evaluation-layer | Future artifact dependency |

Any later scope change should add or update a row here first, then update the
module docstring or issue that implements the change.
